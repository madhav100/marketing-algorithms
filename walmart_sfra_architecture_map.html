This page is a study-focused map of an SFRA-style storefront structure, showing how folders connect from request handling to rendered commerce pages.

Reading tip: the browser calls routes in controllers, controllers use models and scripts, templates render output, and client assets style and enhance the browser experience.

What Each Part Does

package.json

Project metadata and local tooling entry point.

  • Can hold local scripts for build, lint, or asset tasks
  • Supports local developer workflow
Project setup

cartridge/

Main storefront code container.

  • Groups routes, models, scripts, templates, forms, and assets
  • Acts as one deployable storefront unit in SFRA thinking
Storefront unit

client/

Browser-facing assets.

  • default/ holds general static client assets or shared front-end structure
  • js/ contains front-end interactions like click handlers, AJAX, mini-cart behavior
  • scss/ holds styling for layout, buttons, forms, and page design
Browser layer

controllers/

Request and route handling.

  • Home.js serves landing or homepage flows
  • Product.js handles product detail and product-related requests
  • Cart.js manages cart pages and cart actions
Route layer

forms/

Structured form definitions.

  • contactus.xml defines a contact form structure
  • Helps describe expected inputs and validation shape
  • Supports storefront form workflows
Input layer

models/

Data shaping for views.

  • product/ prepares product-related view data
  • account/ prepares customer/account data for rendering
  • Turns raw commerce data into cleaner page-ready structures
View-model layer

scripts/

Reusable server-side helpers and logic.

  • helpers/ contains shared business helpers
  • middleware/ contains reusable request processing steps
  • util/ holds low-level utilities used across the storefront
Reusable backend logic

templates/default/

Display layer for storefront pages.

  • home/ page templates for the homepage
  • product/ templates for product pages
  • cart/ templates for cart pages
  • account/ templates for account pages
Render layer

experience/

Page Designer style content structure.

  • components/ reusable page blocks
  • pages/ page type definitions
  • Supports flexible page assembly by content teams
Experience layer

How the Pieces Connect to Create Commerce

1. Browser request

A shopper opens a route like home, product, or cart.

2. Controller handles the route

Home.js, Product.js, or Cart.js decides what page or action is needed.

3. Scripts and models prepare data

helpers, middleware, utilities, and models collect or shape the needed data.

4. Template renders the page

templates/default turns that data into visible page output.

5. Client assets complete the experience

front-end JavaScript adds interaction and SCSS controls style.

Commerce Example Flows

Homepage flow

Home.js receives the request, homepage data is prepared, the home template renders the page, then client assets enhance banners and navigation.

Product page flow

Product.js handles the request, product models shape data like title, price, images, and availability, then the product template displays the product detail page.

Cart flow

Cart.js manages cart view or updates, scripts help process cart actions, and the cart template displays the current basket experience.

Folder Roles by Layer

Display layer

templates/default, client/js, and client/scss shape what the shopper sees and how the page behaves.

Request layer

controllers receive the request and decide which path to execute.

Logic layer

models and scripts shape and reuse commerce logic so templates stay focused on display.
